Project Journal: Crossing the Database Bridge

Omo, today was intense, but I finally crossed the finish line for Day 9. I successfully built the bridge between my frontend Zustand cart and my backend Postgres database. I created a secure Checkout page UI that dynamically calculates a 7.5% NGN VAT, subtotal, and a flat shipping rate. My biggest achievement was writing a Next.js Server Action using Drizzle ORM to persist my ephemeral client-side cart data directly to the server.

But a lot broke along the way. The biggest roadblock was my database acting like a strict bouncer. When I clicked "Add to Cart," Postgres outright rejected it because of a Foreign Key Constraint—the product variant didn't exist in my inventory yet! I tried using Drizzle Studio to manually seed the database, but my browser kept blocking the local network connection with its security shields. It was exhausting. I finally bypassed the whole headache by writing a "magic bypass" script in my Server Action that automatically generates a dummy category, product, and variant in the background if the database is empty. I also had a minor React panic when a simple variable typo (deliveryFeeKobo instead of shippingKobo) almost crashed my checkout page.

As for what I don't fully understand yet: while my bypass script worked perfectly, writing raw relational database queries with Drizzle still feels a bit like a foreign language. Furthermore, because I chose to park the Supabase authentication for now, I don't yet understand how to merge an anonymous guest's Zustand cart into a real user's Postgres cart when they finally log in. That hybrid state management is still a mystery, but I’m leaving that battle for another day. I built a full-stack cart, and that is a massive win.
